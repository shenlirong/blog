网络协议基础

‌网络协议是计算机网络中用于规定数据通信规则和标准的集合，主要包括TCP/IP、HTTP/HTTPS、UDP、ICMP等核心协议，以及OSI七层模型和TCP/IP四层模型的分层架构。‌

‌核心网络协议‌

  1. ‌HTTP/HTTPS‌。
    • ‌HTTP‌（超文本传输协议）是网页数据传输的基础协议，默认使用80端口，以明文传输数据。
    • ‌HTTPS‌是其安全版本，通过SSL/TLS加密数据，默认使用443端口，适用于敏感信息传输（如支付、登录）。‌‌
  2. ‌TCP/IP协议族‌。
    • ‌TCP‌（传输控制协议）：面向连接，保证数据可靠传输（如文件下载）。
    • ‌IP‌（互联网协议）：负责数据包的路由和寻址。
    • ‌UDP‌（用户数据报协议）：无连接，低延迟，适用于视频流、在线游戏等场景。‌‌
  3. ‌ICMP‌。
    用于网络设备间发送控制消息（如ping命令检测连通性）。‌‌

‌分层模型‌

  1. ‌OSI七层模型‌。
    从下至上包括物理层、数据链路层、网络层、传输层、会话层、表示层、应用层，强调理论划分。‌‌
  2. ‌TCP/IP四层模型‌。
    更贴近实际应用，分为网络接口层、互联网层（IP）、传输层（TCP/UDP）、应用层（HTTP/FTP等）。

‌关键概念‌

  • ‌IP地址与端口号‌：IP标识设备位置（如IPv4的192.168.1.1），端口号标识具体服务（如HTTP的80端口）。
  • ‌DNS‌：将域名（如www.example.com）解析为IP地址。‌
  • ‌NAT‌：将私有IP转换为公网IP，解决IPv4地址不足问题。

http协议基础

‌HTTP协议（Hypertext Transfer Protocol）‌是一个简单的请求-响应协议，通常运行在TCP之上。它定义了客户端和服务器之间的通信规则，如GET、POST等请求机制。HTTP是万维网（WWW）的数据通信基础，用于分布式、协作式和超媒体信息系统的应用层协议‌

HTTP的基本概念

HTTP是一个应用层协议，支持客户端-服务器模型，具有简单、快速、灵活的特点。它通过ASCII格式的请求和响应消息进行通信，消息内容采用类似MIME的格式。HTTP支持多种类型的数据传输，包括文本、图像、音频和视频等，使得网页内容更加丰富多样

HTTP的报文格式

HTTP报文包括请求报文和响应报文：

  • ‌请求报文‌：包含请求行区域（方法、请求URL、版本）、请求首部（可选）和消息体（可选）。例如，GET请求没有消息体，而POST请求有消息体。
  • ‌响应报文‌：包含响应行区域（版本、状态码、原因短语）、响应首部（可选）和消息体（可选）

HTTP的方法

HTTP定义了几种基本的请求方法，常用的有：

  • ‌GET‌：用于请求服务器发送资源，不改变服务器上的资源。
  • ‌POST‌：用于提交要被处理的数据给服务器。
  • ‌PUT‌：用于将资源存储在服务器上。
  • ‌DELETE‌：用于删除服务器上的资源

HTTP的状态码

HTTP响应包括状态码，用于表示请求的结果：

  • ‌2xx‌：成功，如200表示请求成功。
  • ‌3xx‌：重定向，如301表示永久重定向。
  • ‌4xx‌：客户端错误，如404表示未找到资源。
  • ‌5xx‌：服务器错误，如500表示服务器内部错误

http三次握手

‌HTTP的三次握手是TCP协议建立连接的过程，通过客户端与服务器交换三个数据包（SYN、SYN-ACK、ACK）确认双方的收发能力并同步序列号，确保可靠通信‌。

‌三次握手的具体步骤‌

  1. ‌第一次握手（SYN）‌：客户端发送SYN包（SYN=1，seq=x）至服务器，进入SYN_SENT状态，表示请求建立连接。‌‌
  2. ‌第二次握手（SYN-ACK）‌：服务器收到SYN后，回复SYN-ACK包（SYN=1，ACK=1，ack=x+1，seq=y），进入SYN_RECV状态，确认客户端的请求并同步自身序列号。‌‌3‌‌4
  3. ‌第三次握手（ACK）‌：客户端发送ACK包（ACK=1，ack=y+1，seq=x+1），双方进入ESTABLISHED状态，连接建立完成。‌‌‌‌5

‌三次握手的必要性‌

  • ‌避免历史连接干扰‌：若客户端因网络延迟重发SYN，两次握手可能导致服务器误判为新的连接请求，三次握手通过客户端最终确认避免此问题。‌‌5
  • ‌双向能力验证‌：三次握手确保双方均能正常收发数据（客户端通过ACK验证服务器响应能力，服务器通过SYN-ACK验证客户端请求能力）。‌‌2‌‌

‌技术细节补充‌

  • ‌序列号同步‌：三次握手通过交换初始序列号（ISN）确保数据包按序传输，防止乱序或重复。‌‌1‌‌6
  • ‌状态转换‌：客户端从CLOSED→SYN_SENT→ESTABLISHED，服务器从LISTEN→SYN_RECV→ESTABLISHED。‌‌4‌‌7

一句话总结:​ 三次握手就是通信双方互喊 👉 ​​“在吗？”​​ → ​​“在的，你听得到吗？”​​ → ​​“听得到，开始传输吧！”​​
少一次可能“对牛弹琴”，多一次浪费资源，三次是工程最优解！

http 四次挥手

‌HTTP中的四次挥手‌是指TCP协议用来关闭连接的过程，确保两个主机之间能够安全、可靠地关闭连接。四次挥手通过四次消息交换，确保双方都接收到对方的关闭请求，并且数据传输已经完成‌

四次挥手的具体过程

  1. ‌第一次挥手‌：客户端决定关闭连接，向服务器发送一个FIN报文，表示客户端已经完成数据发送，准备关闭连接。服务器收到后回复一个ACK报文，表示确认收到客户端的关闭请求
  2. ‌第二次挥手‌：服务器确认收到客户端的关闭请求后，如果还有数据需要发送给客户端，则发送一个FIN报文。客户端收到后回复一个ACK报文，表示确认收到服务器的关闭请求
  3. ‌第三次挥手‌：服务器发送一个FIN报文，表示服务器已经完成数据发送，准备关闭连接。客户端收到后回复一个ACK报文，表示确认收到服务器的关闭请求
  4. ‌第四次挥手‌：客户端收到服务器的FIN报文后，回复一个ACK报文，表示确认关闭连接‌

四次挥手的作用

  1. ‌确保数据传输完成‌：通过FIN和ACK报文，确保双方都完成了数据传输，没有未发送的数据
  2. ‌防止资源泄漏‌：通过四次挥手，双方可以安全地关闭连接，释放占用的资源，避免资源泄漏
  3. ‌确保连接关闭的可靠性‌：通过多次确认，确保连接关闭的过程是可靠的，避免因网络问题导致连接未正确关闭

浏览器请求数据的过程

1.请求报文：请求行+请求头+空行+请求体
2.响应报文：响应行+响应头+响应体

ReqestHeaders请求头
ResponseHeaders响应头

在HTTP请求中，头部（Header）字段用于携带关于请求或响应的元数据。这些头部字段可以分为几个主要类别，包括通用头部、请求头部、响应头部、实体头部（用于承载请求或响应的实体体，例如JSON或HTML文档）以及表示消息正文的编码和其他特征的扩展头部。下面是一些常见的HTTP请求头部字段及其含义：

通用头部（General Header）

  • Connection: 控制是否需要持久连接。

  • Date: 请求发送的日期和时间。

  • Upgrade: 询问服务器是否愿意升级到其他协议。

请求头部（Request Header）

  • Host: 指定要请求的主机名和端口号（如果有指定）。

  • User-Agent: 发送请求的客户端信息，常用于追踪用户代理。

  • Accept: 浏览器或其他客户端可接受的内容类型。

  • Accept-Language: 浏览器或其他客户端可接受的语言。

  • Accept-Encoding: 浏览器或其他客户端可接受的内容编码。

  • Accept-Charset: 浏览器或其他客户端可接受的字符集。

  • If-Modified-Since: 如果请求的资源自给定时间后未被修改，则返回304状态码。

  • If-None-Match: 如果资源自上次访问后未改变，则返回304状态码。

  • Authorization: 包含认证信息，如Basic认证。

  • Cookie: 向服务器发送Cookie数据。

响应头部（Response Header）

  • Content-Type: 响应体的MIME类型。

  • Content-Length: 响应体的长度（字节）。

  • Location: 重定向的URL。

  • Set-Cookie: 服务器向客户端发送的Cookie。

  • WWW-Authenticate: 服务器对客户端的认证请求。

实体头部（Entity Header）

这些头部字段描述了资源的主体部分，例如：

  • Content-Type: 实体的MIME类型。

  • Content-Length: 实体的长度（字节）。

  • Content-Encoding: 实体的编码方式，如gzip或deflate。

  • Content-Language: 实体的自然语言。

扩展头部（Extension Header）

这些是开发者自定义的头部，用于传递额外的信息，例如：

  • X-Requested-With: 通常由AJAX请求发送，值为XMLHttpRequest。

  • X-Frame-Options: 控制网页是否可以在iframe中显示。

示例

一个简单的HTTP GET请求可能包含以下头部：

GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/58.0.3029.110 Safari/537.36
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,/;q=0.8
Accept-Language: en-US,en;q=0.5
Accept-Encoding: gzip, deflate, br
Connection: keep-alive

这个请求包含了主机名、用户代理、接受的媒体类型、语言、编码以及连接类型等头部信息。理解这些头部字段对于构建有效的HTTP请求和响应至关重要。

响应码

状态码类型描述
200 OK2xx 成功请求成功，并返回所请求的数据。
201 Created2xx 成功请求成功且服务器创建了新的资源（常用于 POST 请求）。
204 No Content2xx 成功请求成功，但没有返回任何内容，常见于删除或更新操作后。
301 Moved Permanently3xx 重定向资源已被永久移动到新 URL，客户端和搜索引擎将更新访问链接。
302 Found3xx 重定向请求的资源临时位于其他 URL，服务器提示继续使用原 URL 访问。
304 Not Modified3xx 重定向缓存内容未修改，无须重新传输数据。
400 Bad Request4xx 客户端错误请求格式错误或缺少必需参数，服务器无法理解请求。
401 Unauthorized4xx 客户端错误请求未经授权，需要提供有效的身份验证凭证。
403 Forbidden4xx 客户端错误服务器理解请求但拒绝执行，可能由于权限限制。
404 Not Found4xx 客户端错误请求的资源不存在，常见于错误链接或资源被删除的场景。

503 Service Unavailable5xx 服务器错误服务器当前无法处理请求，可能由于过载或正在维护中。
